# Prompt v2 Design

Prompt v2 是 RepoPilot 的第二代 Repository Summary Prompt。

相比 Prompt v1，它不再定位为 Repository Summary，而定位为 **AI Onboarding Guide**。

目标不是介绍仓库，而是帮助开发者快速完成项目入职（Onboarding）。

本文档为 Prompt v2 的**设计说明**，将作为未来 `PromptBuilder` 实现的设计依据。本文档不包含实际 Prompt 文本。

相关文档：[prompt-history.md](./prompt-history.md)、[benchmark/evaluation-guide.md](./benchmark/evaluation-guide.md)、ALTER-031（Prompt Engineering Workflow）。

---

# 1. Design Goal

Prompt v2 的设计目标是：让 RepoPilot 的输出从「仓库说明」升级为「入职指南」，帮助开发者在最短时间内建立对项目的**可行动理解**。

RepoPilot 的 Summary 应帮助开发者：

- **快速理解项目定位** — 知道这个项目解决什么问题、处于什么生态位置
- **判断项目是否值得深入阅读** — 明确适用场景与边界，避免无效投入
- **建立整体架构认知** — 理解模块分层、关键目录与数据/控制流向
- **找到最佳阅读入口** — 知道从哪几个文件或模块开始读最高效
- **降低阅读源码成本** — 减少盲目翻目录与重复阅读 README 的时间
- **缩短 Onboarding 时间** — 像有新同事带路一样，而不是独自摸索

**Prompt v2 的目标是：**

帮助开发者**像新加入团队一样理解项目**，而不是简单总结仓库。

# 2. AI Role

## 2.1 身份定义

Prompt v2 中的 AI **不是**通用的 AI Assistant，而是：

**Senior Software Engineer（资深软件工程师）**

## 2.2 职责

AI 的职责是：**帮助一位刚加入团队的新同事理解项目。**

这意味着输出应体现资深工程师的视角：

- 知道哪些信息对新人最重要
- 能区分「必须知道」与「可以稍后了解」
- 能解释技术选型背后的原因，而不只是罗列名称

## 2.3 输出风格

AI 应采用资深工程师的思考方式。输出内容要：

| 要求 | 说明 |
|------|------|
| **解释原因** | 说明「为什么这样设计 / 为什么用这个技术」 |
| **说明价值** | 告诉读者「读完这部分你能得到什么」 |
| **提供建议** | 给出可执行的阅读顺序与行动建议 |

**避免：** 简单堆砌信息、复制 README 原文、无重点的目录罗列。

# 3. Core Principles

Prompt v2 的设计与输出必须遵循以下核心原则：

| 原则 | 含义 |
|------|------|
| **Explain, Don't Enumerate** | 不要只做清单式列举。每一项技术、依赖、目录都应附带「它做什么、为什么重要」。枚举是 v1 的主要问题，v2 必须显著改善。 |
| **Prioritize Information** | 信息必须分优先级。核心模块、关键依赖、必读路径应排在前面；次要细节可压缩或省略。帮助读者在有限时间内获得最大收益。 |
| **Actionable Guidance** | 输出必须可行动。阅读路线、不同目标的推荐路径、建议如何开始 — 都应让读者知道「下一步做什么」，而非停留在抽象描述。 |
| **No Guessing** | 严禁编造。仓库知识（`RepositoryKnowledge`）中未体现的内容，必须明确说明「仓库中未体现」或「无法从现有信息判断」，不得臆测实现细节。 |
| **Chinese First** | 面向中国开发者，正文使用简体中文。项目名称、库名、框架名、代码标识符保持英文，不翻译。 |
| **Repository-aware** | 输出必须基于当前仓库的具体情况。避免通用套话（如「这是一个优秀的开源项目」），应结合该仓库的真实结构、依赖与文档。 |
| **Knowledge-driven** | 输出仅基于 `RepositoryKnowledge` 中已提取的信息（metadata、documents、tree、languages、dependencies、entry points 等）。不假设未索引的源码内容。 |

# 4. Output Structure

Prompt v2 要求 LLM **固定输出**以下 Markdown 章节（按顺序）。章节标题使用 `##`，与 v1 部分标题可复用名称，但**内容与深度要求不同**。

---

## 4.1 项目简介

**章节标题：** `## 项目简介`

**要求：**

- 两段以内
- 必须回答：
  - **项目是什么**
  - **为什么存在**（解决什么问题）
  - **核心价值**（对使用者 / 开发者的价值）

**禁止：** 大段复制 README 首段；空泛形容词堆砌。

---

## 4.2 项目适合谁

**章节标题：** `## 项目适合谁`（**v2 新增**）

**要求：**

必须回答：

- 适合哪些类型的开发者（如后端、全栈、AI 应用开发者）
- 适合哪些岗位场景（如入职上手、贡献开源、课程学习）
- **是否建议新人阅读**（及原因）
- **是否适合作为源码学习项目**（及原因）

**目的：** 帮助读者在 30 秒内判断「这个项目与我是否相关」。

---

## 4.3 技术栈

**章节标题：** `## 技术栈`

**要求：**

- 不仅列举技术名称
- 每项说明：
  - **作用** — 在该项目中承担什么角色
  - **职责** — 负责哪一层能力
  - **为什么使用** — 结合项目类型解释选型合理性（基于已知信息，不猜测）

**禁止：** 仅输出「Python、FastAPI、Docker」式清单。

---

## 4.4 核心依赖

**章节标题：** `## 核心依赖`

**要求：**

- 按**重要程度**排序，而非 `package.json` / `requirements.txt` 文件顺序
- 每个依赖说明：**负责什么**、与项目核心能力的关系
- 区分「核心依赖」与「可略读的依赖」

---

## 4.5 项目结构

**章节标题：** `## 项目结构`

**要求：**

- **不是**简单复述 file tree
- 对每个重要目录 / 模块说明：
  - **职责** — 该目录做什么
  - **重要程度** — 高 / 中 / 低（或等价表述）
  - **建议阅读程度** — 必读 / 选读 / 可跳过（及简要原因）

**依据：** `RepositoryKnowledge` 中的 tree、entry points、documents。

---

## 4.6 推荐阅读路线

**章节标题：** `## 推荐阅读路线`

**要求：**

每一步必须包括：

| 字段 | 说明 |
|------|------|
| **阅读对象** | 具体文件、目录或文档 |
| **为什么** | 为什么先读它 |
| **预计耗时** | 如「约 5 分钟」「约 15 分钟」（合理估计） |
| **阅读收益** | 读完后应理解什么 |

**目的：** 形成可执行的 Onboarding 路径，而非「建议阅读 README」式空建议。

---

## 4.7 不同目标推荐

**章节标题：** `## 不同目标推荐`（**v2 新增**）

**要求：**

针对以下四种目标，分别给出**差异化**推荐阅读路径（可引用 4.6 中的步骤，但侧重点不同）：

| 目标 | 说明 |
|------|------|
| **如果你想：快速使用** | 如何最快跑起来、理解配置与入口 |
| **如果你想：参与开发** | 应熟悉哪些模块、测试与贡献入口 |
| **如果你想：研究架构** | 分层、模块边界、扩展点 |
| **如果你想：学习源码** | 适合精读的文件与阅读顺序 |

---

## 4.8 总结

**章节标题：** `## 总结`

**固定四部分**（建议使用小标题或清晰段落）：

1. **一句话定位** — 用一句话概括项目
2. **最大优点** — 基于仓库信息的客观优点
3. **需要注意** — 复杂度、学习曲线、文档缺口等（不猜测）
4. **建议如何开始** — 一条明确的「第一步」行动建议

# 5. Writing Rules

Prompt v2 输出须遵守以下写作规范：

| 规则 | 说明 |
|------|------|
| **保持 Markdown** | 使用标准 Markdown；标题层级清晰；列表、加粗合理使用 |
| **中文说明** | 叙述性内容使用简体中文 |
| **代码保持英文** | 项目名、库名、路径、标识符不翻译 |
| **不要复制 README** | 必须用自己的话概括；不得大段粘贴 README |
| **不要重复** | 同一信息不在多个章节重复堆砌 |
| **不要罗列** | 避免无解释的 bullet 清单 |
| **多解释 Why** | 技术选型、阅读顺序、目录重要性均需说明原因 |
| **多解释价值** | 告诉读者「读完能得到什么」 |
| **禁止猜测** | 未在 `RepositoryKnowledge` 中出现的实现细节不得编造 |
| **不知道就说明** | 信息不足时写「仓库中未体现」或「无法从现有信息判断」 |
| **保持专业、客观** | 语气像资深同事带新人，不营销、不夸张 |

# 6. Evaluation Checklist

Prompt v2 发布前须通过 Benchmark 评估。以下为设计阶段的检查维度（与 [benchmark/evaluation-guide.md](./benchmark/evaluation-guide.md) 配合使用）。

每项采用 **1–5 分**，满分 **40 分**。

| 评估项 | 分值 | 说明 |
|--------|------|------|
| **准确性** | 5 | 项目定位、技术栈、依赖、结构描述是否与仓库一致；无幻觉 |
| **完整性** | 5 | 是否覆盖 v2 要求的全部章节；关键信息无遗漏 |
| **可读性** | 5 | 结构清晰、段落合理、易于扫读 |
| **Onboarding 价值** | 5 | 是否真正帮助新人上手，而非百科式摘要 |
| **阅读路线质量** | 5 | 路线是否具体、可执行、含 Why / 耗时 / 收益 |
| **是否避免 README 复述** | 5 | 是否独立概括，未复制 README |
| **是否解释 Why** | 5 | 技术、结构、顺序是否说明原因 |
| **中文表达质量** | 5 | 简体中文流畅、专业、无翻译腔 |
| **总分** | **40** | — |

**通过建议：** 单次 Benchmark 平均分 ≥ 32/40，且任一仓库不低于 28/40；与 Prompt v1 对比无显著回归（见 ALTER-025 Prompt Regression Testing）。

# Future Work

Prompt v2 落地后，后续演进方向包括：

| 方向 | 说明 |
|------|------|
| **Prompt v3** | 在 v2 基线上单次引入一项主要改进（如更短输出、更强架构理解） |
| **Architecture Summary** | 独立架构分析 Prompt 与输出结构 |
| **Repository Chat** | 基于 `RepositoryKnowledge` 的多轮问答 |
| **Code Navigation** | 结合 tree 与 entry points 的交互式导航 |
| **Symbol Index** | 符号级索引，减少全文依赖 |
| **Tree-sitter Integration** | 源码解析与符号提取 |
| **Knowledge Cache** | 按 workspace hash 缓存知识，加速重复分析 |
| **Prompt Versioning** | 前端与 API 元数据中的 Prompt Version 与 Benchmark 联动 |
